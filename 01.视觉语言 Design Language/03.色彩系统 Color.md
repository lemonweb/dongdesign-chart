# 色彩系统 Color System

## 1. 文档定位

本文定义图表色彩系统的分层、映射原则和使用规则，参考 Figma 文件 `dongDesign for Chart AI` 中两个 Frame：

- `Basic Colors 基础色板`
- `Semantic Colors 语义色板`

本文回答：

> 面对不同数据关系时，AI 应该如何从基础色板、语义色板和主题变量中选择正确的图表颜色？

本文不维护完整 token 值。变量名、alias 关系、Light / Dark 解析值应查阅 `02.主题变量 Theme-token.md`。

## 2. 与主题变量的关系

`Color.md` 不是变量表，而是色彩决策规则。

```txt
Basic Colors 基础色板
→ Semantic Colors 语义色板
→ Theme Token 主题变量
→ 图表元素绑定
→ Runtime resolved value
```

职责边界：

| 文档 / Frame | 职责 | AI 使用方式 |
|---|---|---|
| `Basic Colors 基础色板` | 定义基础色相、明度、饱和度阶梯 | 用于理解颜色来源和扩展规则 |
| `Semantic Colors 语义色板` | 定义图表数据关系到色彩类型的映射 | 用于判断分类、顺序、发散、状态 |
| `02.主题变量 Theme-token.md` | 记录可绑定变量、alias、Light / Dark 解析值 | 用于生成代码和检查显示值 |
| `03.色彩系统 Color.md` | 定义选择规则、场景判断和禁止项 | 用于 AI 决策和自查 |

AI 生成图表时必须同时保留：

- 绑定变量：例如 `分类色板/item-1`
- 当前模式显示值：例如 light mode resolved value

不得为了显示正确颜色而绕过变量写死色值。

## 3. 色彩系统分层

### 3.1 基础色板 Basic Colors

基础色板是整个图表色彩系统的底层色彩来源。它提供多组色相以及每组 12 阶色值，用于支撑主题、模式和语义色板。

当前基础色板包含：

```txt
blue / cyan / teal / green / lime / yellow / orange / sunset / red / pink / purple / indigo / dark / gray / light
```

基础色板主要承担 4 个作用：

- 为 Light / Dark mode 提供基础色值。
- 为分类色、顺序色、发散色提供可追溯的色彩来源。
- 为新主题扩展提供稳定的色相与阶梯结构。
- 为背景、文字、边框、浮层等结构色提供中性色基础。

使用原则：

- 图表业务元素不应直接引用基础色板。
- 图表业务元素应绑定 `分类色板`、`有序色板`、`发散色板`、`语义色板` 或组件变量。
- 新主题可以调整基础色板，但不应破坏同一色相内部由浅到深的阶梯关系。
- Light / Dark 之间通过 mode token 解析变化，不复制一套图表规则。

### 3.2 语义色板 Semantic Colors

语义色板是图表用色的决策层。它把数据关系转换为可识别的颜色类型。

| 色彩类型 | 数据关系 | 对应变量 | 核心判断 |
|---|---|---|---|
| 分类色板 | 无序类别 | `分类色板/item-*` | 颜色用于区分对象 |
| 顺序色板 | 连续高低、强弱、多少 | `有序色板/*` | 颜色用于表达程度 |
| 发散色板 | 围绕中心值的双向偏离 | `发散色板/*` | 颜色用于表达方向和偏离 |
| 语义色板 | 状态、风险、告警、正常 | `语义色板/*` | 颜色用于表达含义 |
| 组件色 | 轴线、网格、图例、Tooltip、背景 | `坐标轴/*`、`图例/*`、`提示/*`、`背景/*` | 颜色用于支撑阅读 |

## 4. 颜色映射原理

Figma 中的语义色板不是简单的“好看的颜色集合”，而是基于数据关系进行颜色映射。

### 4.1 色相 Hue

色相用于区分不同对象。分类数据应优先通过色相差异建立区分。

适用：

- 不同渠道
- 不同品类
- 不同地区
- 不同用户分群
- 多系列对比

AI 判断规则：

```txt
如果数据项之间没有天然顺序
→ 使用分类色板
```

### 4.2 明度 Lightness / 色调 Tone

明度和色调用于表达同一指标的强弱变化。顺序数据应优先使用同一色相的由浅到深变化。

适用：

- 从低到高
- 从弱到强
- 从少到多
- 密度变化
- 评分等级

AI 判断规则：

```txt
如果数据表达连续程度，且不存在正负中心
→ 使用顺序色板
```

### 4.3 饱和度 Chroma

饱和度用于控制强调程度。越重要、越高强度、越需要关注的值，可以使用更高饱和度或更深色阶。

使用原则：

- 不要只依赖饱和度表达精确数值。
- 高饱和色应优先留给数据系列和状态强调。
- 结构色、网格线、背景应保持低饱和、低对比。

### 4.4 中心值 Center

发散色必须围绕一个明确中心值展开。中心值可以是 `0`、均值、目标值、基准线或业务阈值。

AI 判断规则：

```txt
如果数据同时表达高于 / 低于某个基准
→ 使用发散色板
```

### 4.5 状态含义 Meaning

语义色必须绑定业务含义，而不是绑定数值方向。

例如：

- `normal` 表示正常、健康、通过。
- `warning` 表示临界、预警、需要关注。
- `danger` 表示异常、失败、严重风险。

注意：增长不一定是好事，下降也不一定是坏事。成本上升、投诉上升、错误率上升通常应被判断为风险。

## 5. 分类色板

分类色用于表达“对象不同”，不表达大小强弱。

典型图表：

- 分组柱状图
- 多系列折线图
- 饼图 / 环图
- 分类散点图
- 类目图例

使用规则：

- 优先从 `分类色板/item-1` 开始顺序取色。
- 同一业务对象在同一页面、同一报告或同一看板中必须保持同一颜色。
- 分类数量建议不超过 8 个；超过后应考虑合并、筛选或改用表格。
- 饼图 / 环图建议最多展示 5 个主要分类，其余归入“其他”。
- 分类色之间应保持足够色相差异，避免相邻系列难以区分。

禁止项：

- 不要用分类色表达从低到高。
- 不要用分类色表达 normal / warning / danger。
- 不要在同一页面让同一对象出现多个颜色。

## 6. 顺序色板

顺序色用于表达连续数值的程度变化，通常使用单一色相从浅到深排列。

典型图表：

- 热力图
- 地图
- 矩阵强弱
- 密度分布
- 评分等级
- 日历热力图

使用规则：

- 同一图表中优先使用一个主色相。
- 色阶数量按数据粒度选择，常用 5、7、10 级。
- 必须提供图例，说明最小值、最大值、单位和区间规则。
- 深色通常代表更高、更强或更需要关注，除非业务有明确反向约定。
- 暗色模式下仍绑定同一顺序色变量，由 mode 解析显示值。

禁止项：

- 不要用顺序色表达无序分类。
- 不要在同一个连续指标中混用多个无语义关系的色相。
- 不要省略图例。

## 7. 发散色板

发散色用于表达围绕中心值的双向偏离。

典型场景：

- 正负值
- 增长与下降
- 高于均值与低于均值
- 达标与未达标
- 风险与机会
- 偏离基准线

使用规则：

- 必须先定义中心值，例如 `0`、均值、目标值或行业基准。
- 五色发散适合低粒度判断。
- 十色发散适合连续偏离、热力图、地图。
- 图例必须标注中心值、两端含义和方向。
- 两端颜色必须有明确业务解释，不能只靠红绿暗示好坏。

禁止项：

- 不要在没有中心值的场景使用发散色。
- 不要把发散色当作普通分类色。
- 不要只用颜色表达风险或机会，必须配合文字、图例或 Tooltip。

## 8. 语义色板

语义色用于表达状态和风险，不用于普通数据系列。

当前核心语义：

| 语义 | 变量 | 用途 |
|---|---|---|
| normal | `语义色板/normal` | 正常、健康、通过 |
| warning | `语义色板/warning` | 预警、临界、需关注 |
| danger | `语义色板/danger` | 异常、失败、严重风险 |

使用规则：

- 语义色必须和状态含义绑定。
- 同一页面中相同状态必须使用相同语义色。
- `warning` / `danger` 必须配合文本、图例、标记或 Tooltip 说明。
- 如果指标方向与常识冲突，应以业务含义为准。

禁止项：

- 不要把语义色用于普通类目区分。
- 不要只用颜色表达异常。
- 不要把绿色固定理解为增长好、红色固定理解为下降坏。

## 9. 组件色

组件色用于支撑图表阅读，不应抢占数据系列的视觉优先级。

| 元素 | 优先变量 | 使用原则 |
|---|---|---|
| 坐标轴轴线 | `坐标轴/axisLineColor` | 弱于数据系列 |
| 刻度线 | `坐标轴/tickColor` | 只辅助读数 |
| 分隔线 | `坐标轴/splitLineColor` | 保持低对比 |
| 分隔区域 | `坐标轴/splitAreaColor` | 只用于辅助分区 |
| 坐标轴标签 | `坐标轴/axisLabelColor` | 保证可读，不强调 |
| 图例文字 | `图例/legendTextColor` | 与正文层级一致 |
| Tooltip 背景 | `提示/backgroundColor` | 与浮层一致 |
| Tooltip 轴指示线 | `提示/tooltipAxisColor` | 辅助定位，不强调 |
| 页面背景 | `背景/页面背景` | 承载整体页面 |
| 卡片背景 | `背景/cardColor` | 承载图表容器 |

使用规则：

- 组件色优先绑定组件变量。
- 不要用分类色绘制坐标轴、网格线、背景。
- 不要通过手工透明度修正暗色模式，应调整 mode token。

## 10. 图表场景选色

| 场景 | 数据关系 | 首选色彩类型 | 示例 |
|---|---|---|---|
| 类目比较 | 无序类别 | 分类色板 | 不同渠道销售额柱状图 |
| 多系列趋势 | 无序系列 + 时间 | 分类色板 | 多渠道 GMV 折线图 |
| 占比构成 | 无序类别 | 分类色板 | 品类销售占比环图 |
| 数值强弱 | 连续程度 | 顺序色板 | 区域销售热力图 |
| 地理密度 | 连续程度 | 顺序色板 | 城市订单密度地图 |
| 正负偏离 | 围绕中心值 | 发散色板 | 增长率差异热力图 |
| 目标达成 | 基准偏离 | 发散色板或语义色板 | 目标完成率差异 |
| 状态风险 | 明确状态 | 语义色板 | 活动状态、质量告警 |
| 图表结构 | 阅读辅助 | 组件色 | 坐标轴、图例、Tooltip |

## 11. Light / Dark 使用规则

Light / Dark 不改变图表的色彩语义，只改变同一变量在当前模式下的解析值。

```txt
同一图表元素
→ 绑定同一 theme token
→ 根据当前 mode 得到不同 resolved value
```

使用规则：

- 图表元素始终绑定同一个 theme token。
- Light / Dark 只通过 mode 解析值变化，不改变图表结构。
- 不要为暗色模式复制一套图表规则。
- 如果暗色模式对比不足，应调整 mode token 或 theme alias，而不是在单个图表中写死颜色。
- 语义色在暗色背景上必须检查可读性，必要时配合文字、描边或标记增强识别。

## 12. AI 使用流程

AI 生成图表前，应按以下顺序决策：

```txt
1. 判断数据关系
2. 选择色彩类型
3. 读取 Theme Token 中对应变量
4. 绑定变量到图表元素
5. 解析当前 Light / Dark 显示值
6. 生成图例、Tooltip 或说明
7. 执行色彩自查
```

输出时建议包含：

| 字段 | 示例 |
|---|---|
| 数据关系 | `无序类别` |
| 色彩类型 | `分类色板` |
| 绑定变量 | `分类色板/item-1` |
| 当前模式显示值 | 从 `Theme-token.md` 读取 resolved value |
| 图例要求 | 展示分类名称和对应色块 |
| 风险说明 | 分类超过 8 个时建议合并 |

## 13. 禁止项

- 禁止不判断数据关系就直接取色。
- 禁止用基础色板直接绑定图表业务元素。
- 禁止用分类色表达连续强弱。
- 禁止用顺序色表达无序分类。
- 禁止在没有中心值时使用发散色。
- 禁止把语义色当作普通系列色。
- 禁止只靠颜色表达异常、风险或状态。
- 禁止在图表代码中绕过变量写死色值。
- 禁止为了暗色模式复制一套图表规范。

## 14. AI 自查清单

生成图表后检查：

- 是否先判断了数据关系？
- 是否选择了正确的色彩类型？
- 是否绑定 Theme Token，而不是基础色值？
- 是否读取了当前 mode 的 resolved value？
- 分类色数量是否可读？
- 顺序色和发散色是否提供图例？
- 发散色是否有明确中心值？
- 语义色是否配合文字或 Tooltip？
- 结构色是否弱于数据系列？
- Light / Dark 下是否保持同一变量绑定？
