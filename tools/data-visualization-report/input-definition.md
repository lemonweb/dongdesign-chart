# input-definition · 输入契约

定义本 Skill 接受的**输入形态**，以及如何抽取成 Report IR（见 `report-ir-schema.json`）。

## 一、合法输入

一段**文字版数据解读报告**（Markdown 或纯文本），通常包含：

| 部分 | 说明 | 是否必需 |
|---|---|---|
| 报告信息 | 报告类型、分析对象（商品/业务线…）、SKU/类目、统计周期、状态 | 建议 |
| 核心结论 | 经营表现、问题定位、优化建议（总览） | 建议 |
| 分析模块 | 核心数据 / 趋势 / 流量 / 转化 / 价格 / 竞争… 每块含「诊断」+ 数据（表格/指标/趋势） | 必需（≥1） |
| 明细表 | 多列结构化数据（渠道、关键词、投放、竞品…） | 可选 |
| 优化建议 | 分模块的可执行建议（含预期效果、关注指标） | 可选 |
| 总结 | 全局收敛结论 | 建议 |

## 二、抽取原则

- 逐段识别：标题/周期/对象、每个模块的「诊断」句、指标+数值+环比/同比、明细表、建议。
- **产品判定**：判断该报告属于**对内产品**还是**对外产品**（决定红绿方向，见 visual-style-rules）。若无法判断，默认按对外（红涨绿跌）并在交付时说明。
- **不编造**：源文本没给的数值不虚构；趋势图等需要逐点数据时，只锚定文本给出的关键点（如"6/30 低谷 200"），其余为示意并注明。
- **忠实矛盾**：源文档前后矛盾（如总结与正文口径不符）时，数据模块忠实正文、总结段原样呈现，并向用户指出。

## 三、产出 Report IR

抽取结果组织成 `report-ir-schema.json` 定义的结构（报告元信息、KPI、模块、表格、图表、建议、总结），再进入组件映射与渲染。IR 是"解析"和"渲染"之间的**可校验中间层**——解析错误与渲染错误分开排查，也便于将来替换渲染端（HTML→PDF/PPT）。
