# Dashboard Layout Logic — 看数路径 → 布局逻辑 → 4×N 栅格

> 从真实看板里抽象「看数逻辑」和「布局排版逻辑」，**不按业务场景把布局框死**。目标是沉淀一套可自由组合的布局语言，让 AI 能根据业务诉求自动拼出合适的看板。
>
> 说明：本方法只讨论**图表内容区**，不包含导航栏、菜单、侧栏等框架区域。参考样本均为专业分析师搭建的真实看板。
>
> 这是本 skill 的「空间结构」标准，只负责回答组件为什么放在这里、占多大空间。图表选型、视觉与交互规则从运行时解析到的 DongDesign Chart Wiki 获取，不在本文件重复维护。

## 一、从真实看板抽象「看数路径」

不同业务看板的主题不同，但用户看数时并不是随机浏览图表，而是**沿着一条判断路径逐步建立结论**。

### 样本 → 看数路径（示例，看板名已做代表化处理）

| 看板样本 | 看数路径 |
|---|---|
| 秒杀日报 | 结果总览 → 渠道对比 → 人群拆解 → 趋势复盘 → 归因 → 明细 |
| 标品库数据监控（主站） | 口径说明 → 能力总览 → 应用漏斗 → 类目/事业群拆解 → 覆盖率/属性质量 |
| 发品数据看板 | 口径说明 → 渠道发品趋势 → 渠道占比 → 服务商/类目 TOP → 异常来源解释 |
| 门户 Boss 损益 | 财务口径 → 损益概览 → 趋势 → 报表 → 客户/采销明细 |
| 理赔成本监控（全量） | T-1 核心指标 → 区域/部门成本达成 → 明细表 → 异常部门定位 |
| 秒送体验分析（运营支持版） | 筛选条件 → 履约核心指标 → 大区/城市/商家/距离下探 → 改善趋势 |
| … | … |

### 看数意图 → 核心判断任务

看数意图用于判断用户看这张看板**主要想完成什么任务**。它不是业务分类，也不是固定模板，而是后续拆解看数路径的判断依据。

| 看数意图 | 核心判断任务 |
|---|---|
| 总览 | 建立整体状态判断：当前表现如何，是否正常，重点变化是什么 |
| 分析 | 找到关键结论：发生了什么变化，主要原因可能是什么 |
| 对比 | 判断差异来源：谁高谁低，差距在哪里，由什么维度造成 |
| 监控 | 识别异常和风险：是否异常，严重程度如何，是否需要处理 |
| 漏斗 | 定位转化瓶颈：哪一步流失明显，影响范围和原因是什么 |
| 明细 | 支持对象核查：具体对象是谁，是否需要进一步追踪或处理 |

> 一张看板可以同时包含多个看数意图，但通常会有一个**主意图**。主意图决定核心判断任务，并影响后续看数路径和步骤权重。

### 通用看数步骤

从真实样本的看数路径中，可以抽象出一组通用看数步骤：

```
标题/口径说明 → 核心结果 → 趋势变化 → 维度拆解 → 归因定位 → 对象明细 → 后续动作
```

这不是固定流程（除看板标题外），也不是每张看板都必须完整包含所有步骤。它更像一套**可组合的看数语言**：不同业务只是选择了其中不同步骤，并调整各步骤的权重。

| 看数步骤 | 步骤要回答的问题 |
|---|---|
| 标题/口径说明 | 这个看板是干嘛的，这个数怎么算的？可信吗？ |
| 核心结果 | 现在整体表现怎么样？ |
| 趋势变化 | 最近是变好还是变差？ |
| 维度拆解 | 是哪个渠道/区域/人群/商品导致的？ |
| 归因定位 | 为什么会这样？问题原因是什么？ |
| 对象明细 | 具体是哪批订单/商家/商品/客户？ |
| 后续动作 | 接下来要处理什么？谁负责？ |

> **结论：看板布局的本质不是「业务分类」，而是把看数路径和步骤权重转化为空间结构。**

## 二、从看数逻辑到布局逻辑

看数路径回答的是「先看什么、再看什么」。布局排版回答的是：**哪些信息最重要？哪些应该放首屏？哪些应该放大？哪些只是辅助说明？**

```
业务诉求 → 看数目的 → 看数路径 → 信息优先级 → 布局排版
```

- **业务诉求**：决定这张看板要解决什么问题。
- **看数路径**：决定用户沿着哪些步骤完成判断。
- **步骤权重**：决定这张看板更重在总览、趋势、拆解、归因、明细，还是行动。
- **信息优先级**：决定哪些内容前置、放大，哪些内容作为辅助或后续展开。
- **布局排版**：把信息优先级转化为图表内容区的空间分配。

> 换句话说，布局不是图表的平均铺放，而是**对信息优先级的空间分配**。看数目的决定核心判断任务，核心判断任务决定信息优先级，信息优先级又决定首屏和后续区域怎么排。

搭建时，需要判断看数路径中哪个步骤更重要，并据此决定模块大小和位置：

| 如果看板重在 | 布局上通常意味着 |
|---|---|
| 核心结果 | 把 KPI、核心结论、整体表现放在更靠前的位置 |
| 趋势变化 | 给趋势图更大的横向空间，让时间变化更容易被读出 |
| 维度拆解 | 使用多组并列或分组列，让渠道、区域、人群、商品可以比较 |
| 归因定位 | 保留主分析区，并在旁边或下方放解释、排行、异常来源 |
| 对象明细 | 给表格、列表、明细更大的连续空间 |
| 后续动作 | 将待处理事项、责任对象、异常清单放在路径末端或异常区域旁 |

## 三、4×N 栅格布局

看板图表内容区建议采用 **4×N 栅格**，纵向 N 行取决于画布高度、屏幕比例、内容量等。

为什么是 4 列：3 列常见组合只有 1+2、1+1+1，灵活性偏少；5 列难分主次、也不好对齐；6 列虽然更细，但对普通 BI 看板来说容易太碎。因此 **4×N 更适合作为图表内容区的基础栅格**。

4×N 的价值：让不同看数步骤/模块可以通过**跨列、跨行**形成主次关系。

### 栅栏组合

| 组合 | 用途 | 4 列示意 |
|---|---|---|
| **通栏** | 承载全局信息，让用户先建立整体判断 | `[ ████████ 全局/总览 跨4列 ████████ ]` |
| **主次 3+1** | 主分析 + 辅助解释 | `[ ██████ 主(跨3) ██████ ][ 辅 ]` |
| **主次 2+1+1** | 主分析 + 两块辅助 | `[ ███ 主(跨2) ███ ][ 辅1 ][ 辅2 ]` |
| **对比 2+2** | 同级两组比较 | `[ ███ A(跨2) ███ ][ ███ B(跨2) ███ ]` |
| **对比 1+1+1+1** | 同级四组比较 | `[ A ][ B ][ C ][ D ]` |
| **下探** | 纵向扩展 N 行，把核心结果 → 维度拆解 → 对象明细按路径展开 | `[ ██ 主区 跨列×多行 ██ ][ 辅 ]`<br>`[ ██                ██ ][ 辅 ]` |

### 图表的空间适配性（可拉伸性）

栅格分配不能只看「信息优先级」，还要看**图表本身适不适合被拉宽**。同样是 span，折线图拉到通栏读起来更爽，而环形图拉到通栏只会让一个小圆圈飘在正中间、两侧大片留白。因此把图表按空间适配性分三类：

| 适配类别 | 典型图表 | 可拉伸性 | 栅格规则 |
|---|---|---|---|
| **宽度友好型** | 折线 line、面积 area、柱 bar、条形 bar_h、双轴 dual、指标拆解树 tree | 越宽越好读，横向承载趋势/排序/层级 | 可 span 2–4，行尾靠它拉伸填满 |
| **方形 / 固定比例型** | 环形/饼 pie、玫瑰 rose、雷达 radar、漏斗 funnel、仪表盘 gauge | 有固有长宽比，拉宽只留白 | **span 上限 2，绝不单独通栏(span4)，也不作为行尾拉伸对象** |
| **全宽型** | 指标卡 metric_group、排行 rank、明细表 table | 天然需要整行承载 | 固定通栏 span4 |

**方形图的成对规则**：方形图不能独占一行。用到环形/雷达等方形图时，给它配一个**同维度**的伙伴——通常是条形图(bar_h，看具体量级与排序)或占比明细/排行——两者各 span2 并列凑成一行（`2+2`）。这样既满足「铺满不留白」，又比一个孤零零的环形图更有分析价值：**环看结构占比，条/表看具体贡献与差距**。

> 一句话补充：**「铺满不留白」不等于「把任意图表拉到通栏」**。留白要靠"给方形图配伴"来消除，而不是把方形图本身拉宽。

**同一行等高**：同一行并列的组件必须**等高**，不能一高一低。实现上：说明区(caption)高度统一预留（如固定 2 行），同类图表区高度一致，栅格行采用等高对齐(`align-items:stretch`)。参差不齐的行高会显得杂乱、不专业。

### 步骤权重 → 栅格排布倾向

| 步骤权重 | 栅格排布倾向 |
|---|---|
| 重在核心结果 | 顶部或首屏前段放 KPI、核心结论、整体表现（常配通栏 / 4×KPI 一行） |
| 重在趋势变化 | 趋势图**跨 2–4 列**，保证足够的横向阅读空间 |
| 重在维度拆解 | 使用 2+2、1+1+1+1 或多组并列结构 |
| 重在归因定位 | 主图占较大区域（3+1 / 下探），旁边或下方补充解释、排行、异常来源 |
| 重在对象明细 | 表格或对象列表**跨多列、多行**，形成连续阅读区域 |
| 重在后续动作 | 将待处理对象、责任人、建议动作放在路径末端或异常区域旁 |

> 这个映射**不是固定模板，而是布局判断规则**。最终页面可以有很多组合方式，但每一种组合都应该能解释清楚：**为什么这个信息在这里，为什么它这么大，为什么它比另一个信息更靠前。**

### 真实看板布局范式（参考样本抽象）

对一批真实优秀看板（SaaS / B 端 / 分析型，样本见项目内 `看板布局参考图.md`）做抽象后，它们高度收敛到同一套排版语言，可作为生成时的默认骨架：

1. **KPI 簇置顶**——顶部一行核心指标（metric_group / 通栏），大数字 + 同比 delta，先建立现状判断。
2. **趋势 hero 占宽**——最核心趋势图（line/area）给最大横向空间，跨 2–4 列，常放首屏，可"本期 vs 上期"。
3. **构成小环 + 成对**——占比/构成用 pie（span ≤ 2），右侧或旁边配同维度的横条 bar_h 或明细，凑成 `2+2`，环形绝不单独通栏。
4. **达成率用 gauge 装进固定卡**——目标完成率 / 健康分用仪表盘，独立中卡，不拉伸。
5. **右栏窄列放榜单/日程/动态**——排行、日程、最近记录用窄栏（rank 或列表，span ≤ 2，纵向拉长）。
6. **明细宽表沉底**——交易 / 任务等明细用 table（通栏）置底，含状态 pill、可排序。
7. **类目对比中宽**——少类目竖柱 bar、长名/排名横条 bar_h，中宽（span2），峰值深色标注。
8. **周期切换常态**——图表卡右上常带 Weekly/Daily/Monthly 或 当日/当月/当年。

> 默认节奏：**上 KPI → 中（趋势宽 / 对比中 / 构成小且成对）→ 右栏榜单/日程 → 底部宽表**，全程环形/雷达/仪表盘等方形图不通栏。

## 四、AI 如何应用（生成规则）

生成看板布局时，按以下顺序推导，而不是套用固定模板：

1. **识别主看数意图**（总览/分析/对比/监控/漏斗/明细）→ 确定核心判断任务。
2. **规划看数路径**：从「标题/口径 → 核心结果 → 趋势 → 拆解 → 归因 → 明细 → 动作」中选取本次需要的步骤，并给出**步骤权重**（哪一步最重）。
3. **转化为信息优先级**：最高优先级 → 首屏/放大/靠前；辅助 → 旁侧/下方/后续展开。
4. **落到 4×N 栅格**：用「通栏 / 主次 / 对比 / 下探」组合把优先级映射成跨列跨行；标题/口径通常通栏置顶，核心结果常为一行 4 个 KPI 或通栏，趋势跨 2–4 列，拆解用 2+2 或 1+1+1+1，明细跨多列多行，动作/异常置末端。
   - **注意图表可拉伸性**（见上「图表的空间适配性」）：环形/饼/玫瑰/雷达/漏斗/仪表盘等方形图 span 最多 2、绝不单独通栏，必须配一个同维度的条形图或明细凑成 `2+2`；只有宽度友好型(趋势/柱/条形/拆解树)和全宽型(KPI/排行/明细表)才拉到 3–4 列。
5. **每个模块都要可解释**：能说清「为什么在这里、为什么这么大、为什么比别的靠前」。
6. **套用视觉与图表规范**：先按 `SKILL.md` 的 DongDesign Wiki 流程解析本地仓库或 GitHub 回退，再读取选型、视觉语言、交互与引擎适配页面。

> 一句话：**看数意图 → 看数路径 → 步骤权重 → 信息优先级 → 4×N 栅格**，最后用飞哥的图表可视化能力把自然语言转成图表，套进这套布局里。

### 与 Dashboard Spec 的对应字段（建议）

在 Dashboard Spec 里显式记录布局推导，便于校验与复盘：

- `layout.primaryIntent`：主看数意图（总览/分析/对比/监控/漏斗/明细）
- `layout.readingPath`：本次采用的看数步骤有序数组
- `layout.stepWeights`：各步骤权重（如 `{核心结果: high, 趋势变化: mid, ...}`）
- `layout.grid`：`"4xN"`；各 section 标注 `span`（跨列 1–4）与 `rows`（跨行）及组合类型（通栏/主次/对比/下探）
- 每个 section 附 `whyHere`：解释其位置与大小的理由（可解释性要求）
