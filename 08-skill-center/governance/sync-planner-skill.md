# Sync Planner Skill

## 1. 文档定位

本 Skill 用于规划 Wiki 改动向外部系统的同步流程，包括 Figma、Joyspace、GitHub Pages、MCP、脚手架等。

它回答：

> 这一批改动应该按什么顺序、什么粒度、推送到哪些下游系统？需要哪些先决条件？

本 Skill 不直接执行同步，只输出同步计划与发布检查清单。

## 2. 触发场景

- “这批改动准备发布，帮我列计划”
- “这次涉及 Figma 与 Joyspace 同步，先排个顺序”
- “Pages 上线前需要做哪些检查”
- 由 `wiki-maintainer-skill` / `dependency-checker-skill` 在内部修复完成后调用

## 3. 输入要求

| 输入 | 是否必需 | 说明 |
|---|---|---|
| 改动清单 | 必需 | 文件或区块级别 |
| 目标系统 | 必需 | Figma / Joyspace / GitHub Pages / MCP / 脚手架 / 内部库 |
| 时间窗口 | 推荐 | 是否处于冻结期或封版期 |
| 通知对象 | 推荐 | 设计、研发、运营、AI 工具链 owner |

## 4. 必读 Wiki 文件

```txt
README.md
ai-file-index.md
00-overview/how-to-use.md
07-document-governance/metadata-naming-guide.md
08-skill-center/registry.md
08-skill-center/dependency-map.md
```

如仓库根目录存在 `INSTALL.md`、`KNOWLEDGE-UPDATE.md`、`SOP.md` 等流程文档，应一并读取。

## 5. 执行流程

```txt
1. 汇总改动清单与依赖检查结果
2. 按目标系统拆分同步包：内容包 / 资源包 / 索引包 / 配置包
3. 对每个包列出先决条件（文件齐备、依赖通过、命名一致）
4. 安排同步顺序：上游索引 → 内容文档 → 外部系统 → 通知
5. 列出回滚策略与失败处理
6. 输出 Sync Plan，标注负责人占位
```

## 6. 同步维度

| 维度 | 判断点 |
|---|---|
| 范围 | 改动是否影响外部链接、模板或脚手架 |
| 顺序 | 索引、依赖、内容、外部系统的发布顺序 |
| 风险 | 是否处于冻结期或与他人改动冲突 |
| 回滚 | 是否定义了回滚路径 |
| 通知 | 是否覆盖了所有受影响方 |
| 验证 | 同步后如何验证内容生效 |

## 7. 输出格式

```txt
## 同步概览
改动范围：
目标系统：
时间窗口：

## 同步包
- 包 1（内容）：
  目标：
  先决条件：
  操作步骤：
  验证方式：
- 包 2（索引）：
- 包 3（外部系统）：

## 顺序与依赖
1. ...
2. ...
3. ...

## 风险与回滚
- 风险：
- 回滚步骤：

## 通知清单
- 角色：
  通知内容：
  通知方式：

## 后续跟进
- 待办：
- 责任人占位：
```

## 8. 依赖 Skill

| Skill | 关系 |
|---|---|
| `wiki-maintainer-skill` | 上游，提供改动总览 |
| `dependency-checker-skill` | 提供同步前的依赖检查结论 |
| `design-spec-to-wiki-skill` | 当本次同步涉及设计规范回填时联动 |

## 9. 不负责事项

- 不直接执行同步命令。
- 不修改外部系统配置。
- 不替代图表层或代码层的评审。
- 不发起非本仓库的内容变更。

## 10. 自查清单

- 是否覆盖全部目标系统？
- 是否给出顺序与先决条件？
- 是否定义回滚与验证方式？
- 是否标注通知对象？
- 是否说明冻结期 / 风险窗口？
- 是否区分了内容 / 索引 / 资源同步包？
