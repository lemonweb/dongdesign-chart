# Markdown Parser Rules

## 1. 解析目标

将 Markdown 从“文本排版结构”解析为“经营报告语义结构”。不要保留 Markdown 的表层顺序作为最终报告顺序。

必须抽取：

- 报告标题、对象、时间周期、报告类型。
- 章节层级与每个章节的目的。
- 指标名称、数值、单位、变化率、正负向、严重程度。
- 表格列、行、维度、度量。
- 结论、原因判断、行动建议。
- 每条结论和建议的证据来源。

## 2. 标题与元信息

| 原文线索 | IR 字段 |
|---|---|
| H1 标题 | `report_meta.title` |
| 商品/店铺/活动名称 | `report_meta.report_object` |
| 日期、周期、统计范围 | `report_meta.period` |
| “单品”“活动”“转化”“流量”等关键词 | `report_meta.report_type` |
| “给老板看”“运营复盘”“商家报告”等 | `report_meta.audience` |

缺失时写 `unknown`。从标题推断时标注 quality flag。

## 3. 指标抽取

识别以下指标模式：

```txt
指标名 + 数值 + 单位
指标名 + 同比/环比/较上期 + 变化率
上升/下降/增长/下滑/持平 + 数值
金额、访客、点击、转化、ROI、客单价、加购、收藏、退款等业务指标
```

每个指标必须生成：

| 字段 | 规则 |
|---|---|
| `id` | 使用稳定小写英文或拼音短名，如 `gmv`, `uv`, `conversion_rate` |
| `name` | 保留原文指标名 |
| `value` | 保留原文数值格式；能可靠转换时可补充数值 |
| `unit` | 元、%、人、次、件、unknown |
| `change` | 同比/环比/较上期变化 |
| `change_type` | `wow` / `mom` / `yoy` / `period_over_period` / `unknown` |
| `polarity` | 对经营结果有利为 positive，不利为 negative |
| `severity` | 按变化幅度和业务上下文判断 |
| `source_ref` | 指向原文段落或表格 |

## 4. 表格解析

表格先作为数据容器进入 IR，不默认作为最终视觉表达。

| 表格类型 | 识别线索 | 后续建议 |
|---|---|---|
| KPI 表 | 指标、数值、变化率 | 转 KPI 卡组 |
| 时间序列表 | 日期、小时、周、月 | 转趋势图 |
| 类别排名表 | 渠道、商品、关键词、地域 | 转横向条形图 |
| 竞品对比表 | 本品、竞品、价格、销量 | 转对比矩阵/排名条 |
| 转化链路表 | 曝光、点击、访问、加购、成交 | 转漏斗/路径图 |
| 明细表 | 订单、SKU、ID、记录列表 | 可保留为表格 |

## 5. 结论与建议抽取

识别具有判断性的句子：

- “主要问题是...”
- “原因可能是...”
- “建议优先...”
- “需要关注...”
- “风险在于...”

结论必须绑定证据：

```txt
claim → evidence_refs → source_ref
```

找不到证据的结论不得写为确定结论，只能进入 `quality_flags` 或 `low confidence insight`。

## 6. 归因处理

允许输出归因线索，不允许输出无依据归因。

| 情况 | 处理 |
|---|---|
| 原文明确给出原因与数据 | 写入 insight，confidence high |
| 原文有数据但未明确原因 | 写“可能关联”，confidence medium |
| 原文无数据支撑 | 写入风险，不作为结论 |

## 7. 冲突与模糊内容

- 日期冲突：保留全部值，写入 `source_trace.conflicts`。
- 小字/OCR 模糊：标注 `low_confidence_ocr`。
- ID 或 SKU 不确定：不得自行修正。
- 数值单位缺失：单位写 `unknown`。
- 表格列错位：保留原文并标注解析风险。
