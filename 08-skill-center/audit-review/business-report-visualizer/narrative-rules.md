# Narrative Rules

## 1. 叙事目标

将原始报告从“信息罗列”重组为“经营诊断故事线”。

默认故事线：

```txt
1. 发生了什么？
2. 问题有多严重？
3. 问题主要出在哪里？
4. 哪些因素最可能导致问题？
5. 接下来优先做什么？
```

## 2. 标准叙事结构

| 阶段 | 作用 | 推荐组件 |
|---|---|---|
| 开场结论 | 一句话说明经营状态 | `InsightSummary` |
| 核心异动 | 用 3~5 个关键指标证明问题 | `KpiGroup` |
| 趋势证据 | 说明问题从何时开始 | `TrendChart` |
| 结构拆解 | 渠道/搜索/价格/转化/竞品定位 | `BreakdownChart` / `ComparisonMatrix` |
| 原因判断 | 收敛 2~3 个关键原因 | `DiagnosisCardGroup` |
| 行动建议 | 按优先级给可执行动作 | `ActionPlan` |

## 3. 重组规则

- 不照搬原始 Markdown 的章节顺序。
- 优先把结论和核心指标放到首屏。
- 把“指标变化”放在“原因判断”之前。
- 把“建议”与对应证据绑定，不孤立展示。
- 删除重复铺陈，但不得删除原始数据事实。
- 当报告没有明确主结论时，从 key metrics 中生成低置信开场，并标注 `inferred`。

## 4. 结论文案规则

开场结论必须符合：

```txt
经营对象 + 核心变化 + 影响/风险 + 优先关注方向
```

示例：

```txt
冰岛普洱生茶本周期流量与成交同步下滑，搜索承接和价格竞争力是优先排查方向。
```

不得使用：

- “整体表现良好”这类无证据泛化。
- “一定是因为...”这类过度归因。
- “爆款”“崩盘”等夸张表达，除非原文和数据明确支撑。

## 5. 严重程度判断

| severity | 参考规则 |
|---|---|
| `critical` | 核心指标下滑超过 40%，或转化链路断点明确 |
| `high` | 核心指标下滑 20%~40%，且多个指标同向恶化 |
| `medium` | 单一关键指标波动 10%~20%，或影响范围局部 |
| `low` | 轻微波动或数据不足 |
| `neutral` | 无明显风险或正负混合 |
| `unknown` | 数据不足 |

不同业务指标阈值可能不同；当 Wiki 或用户口径给出阈值时，以业务口径为准。

## 6. 行动建议规则

行动建议必须包含：

| 字段 | 说明 |
|---|---|
| 建议标题 | 动词开头，面向执行 |
| 优先级 | high / medium / low |
| 预期影响 | 指向指标改善，不夸大 |
| 证据 | 至少 1 条 source_ref |
| 责任提示 | 运营、投放、商品、设计、研发等 |

建议排序：

```txt
高影响 + 低成本
→ 高影响 + 中成本
→ 中影响 + 低成本
→ 需要更多数据验证
```

## 7. 缺口表达

当缺少数据或证据时，使用清晰风险表达：

```txt
当前报告没有给出搜索关键词排名变化，无法确认搜索承接是否为主因。
```

不要把缺口隐藏在设计稿或图表里。
