# 核心规则写法原则 Core Rules Writing Guide

## 1. 文档定位

本文定义 `05-rules/` 中 AI 执行规则、生成流程和禁止项文档的写法。

核心规则文档负责约束 AI “按什么顺序做、必须读什么、什么行为禁止、什么情况下不能交付”。

## 2. 写作目标

核心规则文档应回答：

> AI 在图表设计、代码生成、审查和维护任务中，应该遵守什么工作流和边界？

## 3. 标准章节

建议结构：

```txt
1. 文档定位
2. 适用范围
3. 执行流程
4. 必读文件
5. 分支规则
6. 输出要求
7. 禁止项
8. 阻断条件
9. 自查清单
```

## 4. 流程写法

流程必须写成有顺序的动作。

推荐：

```txt
1. 识别业务问题。
2. 判断数据结构。
3. 读取能力矩阵和选型规则。
4. 读取具体图表文档。
5. 生成方案并自查。
```

不推荐：

```txt
AI 应综合考虑并合理生成。
```

## 5. 规则粒度

核心规则应写成可执行判断：

| 类型 | 写法 |
|---|---|
| 必须 | 用于不可跳过的动作 |
| 禁止 | 用于明确不能做的行为 |
| 优先 | 用于存在多个合理方案时的排序 |
| 兜底 | 用于 Wiki 缺口下的临时处理 |
| 阻断 | 用于不能声明交付完成的条件 |

## 6. 与其他文档的关系

核心规则不重复写具体样式值，而是要求读取：

- `01-design-language/`：视觉语言。
- `02-chart-type/`：具体图表组件。
- `04-adaptation/`：代码与引擎映射。
- `06-self-check/`：交付前检查。
- `08-skill-center/`：任务执行 Skill。

## 7. 禁止项

- 禁止把核心规则写成抽象原则而没有执行步骤。
- 禁止跳过必读文件。
- 禁止把兜底实践写成正式规范。
- 禁止让 AI 在缺字段、缺口径时直接定稿。
- 禁止只描述“应该好看”，不定义检查方式。

## 8. 自查清单

- 是否有明确适用范围？
- 是否有可执行的步骤顺序？
- 是否列出必读文件？
- 是否说明分支和兜底规则？
- 是否明确禁止项和阻断条件？
- 是否不重复定义具体样式规范？
