# 通用写作原则 General Writing Principles

## 1. 文档定位

本文定义 dongDesign Chart Wiki 的通用写作原则，适用于所有 Knowledge Layer 和 Execution Layer 文档。

目标是让文档同时服务人类阅读和 AI Agent 执行：内容应清晰、可检索、可引用、可验证。

## 2. 基本原则

| 原则 | 写法要求 |
|---|---|
| 先定位 | 每篇文档必须说明自己负责什么、不负责什么 |
| 可执行 | 规则应能指导设计、代码、审查或维护动作 |
| 可判断 | 避免“适当、美观、合理”等无法判断的词 |
| 可追溯 | 涉及上游规则时引用具体文件 |
| 不重复 | 单个文档不重复定义全局色彩、字体、轴、图例等规则 |
| 不推断成规范 | 缺少证据时标记 `inferred` 或 `needs_spec` |

## 3. 推荐结构

通用文档优先使用以下结构：

```txt
1. 文档定位
2. 使用时机
3. 核心规则
4. 与其他文档的关系
5. AI 使用规则
6. 禁止项
7. 自查清单
```

具体类型文档可按对应写作指南调整。

## 4. 语言风格

- 使用中文标题和说明，路径、字段、变量、配置项使用英文或代码格式。
- 每条规则优先写成“条件 + 动作 + 风险”。
- 表格用于规则对比、字段定义、状态判定和映射关系。
- 列表用于步骤、禁止项和自查项。
- 不写纯概念百科，必须落到 AI 或设计/工程动作。

## 5. 引用规则

引用其他文档时使用稳定相对路径，例如：

```markdown
读取 `01-design-language/color.md`。
```

不得引用不存在的路径或迁移前中文路径。新增或移动文件后必须同步更新：

- `ai-file-index.md`
- 目录 `README.md`
- 相关 Skill 必读路径
- `08-skill-center/dependency-map.md`（如涉及 Skill 依赖）

## 6. 状态标记

当规则来源不完整时，必须标记状态：

| 状态 | 含义 | 使用方式 |
|---|---|---|
| `confirmed` | 已有明确 Wiki 规则或用户确认 | 可作为规范输出 |
| `partial` | 有部分依据但缺少完整口径 | 输出时说明限制 |
| `inferred` | 由上下文推断 | 只能作为建议 |
| `needs_spec` | 缺少规范或物料 | 不得写成确定规则 |

## 7. 禁止项

- 禁止复制大段外部原文而不转化为规则。
- 禁止把截图现象直接写成通用规范。
- 禁止在多个文件重复维护同一 token、字号或颜色值。
- 禁止新增文件后不更新索引。
- 禁止把临时兜底方案写成正式规范。

## 8. 自查清单

- 是否说明文档定位和边界？
- 是否包含使用时机？
- 是否能被 AI 执行为具体动作？
- 是否引用了正确上游文件？
- 是否避免了不可判断的模糊描述？
- 是否标记了推断或缺口？
- 是否同步更新索引和 README？
