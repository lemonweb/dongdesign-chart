# 视觉语言写法原则 Design Language Writing Guide

## 1. 文档定位

本文定义 `01-design-language/` 中视觉语言文档的写法。

视觉语言文档负责沉淀所有图表共享的样式规则，包括设计原则、主题变量、色彩、字体、布局、坐标轴、图例、标签和 Tooltip。

## 2. 写作目标

视觉语言文档应回答：

> 当 AI 生成或审查任意图表时，通用视觉元素应该如何选择、布局、绑定 token 和避免误用？

它不负责定义某个具体图表的全部结构。具体图表规则应写入 `02-chart-type/`。

## 3. 标准章节

建议结构：

```txt
1. 文档定位
2. 与主题变量的关系
3. 使用时机
4. 核心规则
5. 场景映射
6. 与图表类型的关系
7. G2 / ECharts 映射提示
8. 禁止项
9. AI 自查清单
```

并非每篇都必须完整使用，但不得缺少定位、核心规则、禁止项和自查清单。

## 4. 样式值写法

样式值应优先通过 token 表达：

```txt
绑定 token → 当前 mode resolved value → 使用场景
```

写法要求：

- 变量名、alias、Light / Dark 解析值写入 `theme-token.md`。
- 决策规则写入对应视觉语言文档，例如 `color.md`。
- 不在多个文档重复维护同一色值或字号。
- 若 token 未定义，标记 `needs_spec`，不得临时创造正式变量。

## 5. 场景规则写法

视觉语言规则应围绕场景写，而不是只列属性。

推荐：

```txt
当图表需要表达无序分类时，使用分类色板，并保持同一业务对象跨页面颜色一致。
```

不推荐：

```txt
蓝色很好看，适合大多数图表。
```

## 6. 与图表类型的关系

视觉语言文档定义通用规则，图表类型文档定义具体使用方式。

例如：

- `legend.md` 定义图例 shape、顺序、布局和状态。
- `line-chart.md` 定义折线图应该使用 line shape 图例。
- `bar-chart.md` 定义柱状图应该使用 rectangle shape 图例。

若具体图表需要例外规则，必须说明原因和影响范围。

## 7. 禁止项

- 禁止把单个图表示例中的局部样式写成全局规则。
- 禁止直接复制图表库默认样式作为 dongDesign 规范。
- 禁止用“美观、高级、突出”等词替代可执行规则。
- 禁止在视觉语言文档中重新定义业务指标口径。
- 禁止新增 token 名称但不说明作用域、mode 和使用场景。

## 8. 自查清单

- 是否说明该视觉元素服务什么读数或判断？
- 是否说明 token 绑定和 resolved value 的关系？
- 是否区分通用规则和图表特例？
- 是否引用相关图表类型文档？
- 是否说明禁止项和误用风险？
- 是否能被 VibeDesign 和 VibeCode 同时使用？
