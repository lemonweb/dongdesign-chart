# AI 执行规则写法原则 AI Rules Writing Guide

## 1. 文档定位

本文定义面向 AI Agent 的规则、Skill、执行流程和输出约束的写法。

AI 规则文档的目标不是给人类做概念介绍，而是让 Agent 能稳定选择文件、执行步骤、输出结果并避免越界。

## 2. 写作目标

AI 规则文档应回答：

> 什么时候触发？必须读什么？输入需要什么？按什么步骤做？输出什么？哪些事不能做？

## 3. 标准结构

建议结构：

```txt
1. 文档定位
2. 触发场景
3. 输入要求
4. 必读文件
5. 执行流程
6. 输出格式
7. 依赖 Skill
8. 不负责事项
9. 自查清单
```

## 4. 触发场景写法

触发场景应写用户意图，而不是工具名称。

推荐：

```txt
- “帮我选一个适合这个数据的图表”
- “检查这个图表是否符合规范”
```

不推荐：

```txt
- 使用某某工具时触发
```

除非该 Skill 明确只服务某个工具。

## 5. 必读文件写法

必读文件必须写精确路径，并按读取顺序排列。

```markdown
1. `ai-file-index.md`
2. `02-chart-type/capability-matrix.md`
3. `02-chart-type/selection-rules.md`
```

不得写“读取相关文档”这种无法执行的描述。

## 6. 输出格式写法

输出格式应给出字段或模板。

示例：

```yaml
chart_type:
primary_intent:
required_fields:
wiki_refs:
risks:
next_action:
```

字段必须服务下游使用，不为形式而列字段。

## 7. 边界写法

每个 AI 规则文档必须说明不负责事项。

示例：

- 不定义业务指标口径。
- 不编造缺失数据。
- 不覆盖上游视觉语言规范。
- 不直接修改未授权文件。

## 8. 禁止项

- 禁止让 AI 跳过必读文件。
- 禁止把输入不足的任务直接定稿。
- 禁止输出没有证据的业务判断。
- 禁止将 Skill 写成厚知识库，复制大量规范正文。
- 禁止多个 Skill 之间职责重叠但不说明优先级。

## 9. 自查清单

- 是否有明确触发场景？
- 是否列出输入要求？
- 是否列出精确必读路径？
- 是否有可执行流程？
- 是否有结构化输出格式？
- 是否说明不负责事项和禁止项？
- 是否引用 `skill-runtime-contract.md` 或相关运行契约？
