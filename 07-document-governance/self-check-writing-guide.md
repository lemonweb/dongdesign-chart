# 示例自查写法原则 Self Check Writing Guide

## 1. 文档定位

本文定义 `06-self-check/` 中检查清单、错误案例和验证示例的写法。

自查文档用于帮助 AI 和人类在交付前发现问题，不负责重新定义完整规范。

## 2. 写作目标

自查文档应回答：

> 交付前需要检查哪些点？哪些问题出现时不能通过？发现问题后应该回到哪个规范修正？

## 3. 检查清单写法

检查项应使用复选框，并写成可判断问题：

推荐：

```markdown
- [ ] 折线图 hover 指示线是否为 `1px` 实线，并限制在绘图区内？
```

不推荐：

```markdown
- [ ] hover 是否好看？
```

每个检查项应尽量对应一个明确规范来源。

## 4. 错误案例写法

错误案例建议包含：

| 字段 | 说明 |
|---|---|
| 问题名称 | 简短描述反模式 |
| 触发场景 | 什么时候容易出现 |
| 错误表现 | 用户或 AI 会看到什么 |
| 影响 | 为什么不能通过 |
| 修正方式 | 回到哪个规则修 |
| 引用规范 | 对应 Wiki 路径 |

## 5. 验证示例写法

验证示例应保留输入、检查过程和结论。

建议结构：

```txt
1. 示例背景
2. 输入图表或代码摘要
3. 使用的规范文件
4. 检查结果
5. 问题列表
6. 修正建议
7. 是否通过
```

## 6. 禁止通过项

自查文档应明确哪些问题不能通过，例如：

- 图表无法回答业务问题。
- 字段映射缺失。
- 使用随机颜色或默认主题。
- 文字重叠、标签溢出或 Tooltip 遮挡主读数。
- VibeDesign 图层不可维护。
- VibeCode 只做语法检查，没有视觉验证。

## 7. 禁止项

- 禁止用自查文档替代上游规范。
- 禁止检查项只写抽象价值观。
- 禁止没有修正路径。
- 禁止为了通过自查而删除必要信息。
- 禁止把单个错误截图当成完整规则来源。

## 8. 自查清单

- 检查项是否可以回答“是 / 否”？
- 是否包含不能通过的阻断项？
- 是否说明发现问题后回到哪个文件？
- 是否覆盖设计、代码、交互和状态？
- 是否避免重复定义上游规范？
