# 图表组件写法原则 Chart Component Writing Guide

## 1. 文档定位

本文定义 `02.图表模式 Chart-type/` 中单个图表组件文档的写法。

图表组件文档用于回答：

> 某一种图表在什么业务目标下适用、需要什么数据结构、由哪些视觉元素构成、有哪些交互规则，以及 AI 在生成设计或代码时必须遵守哪些限制。

本文不定义具体图表的视觉样式值。颜色、字体、坐标轴、图例、标签、Tooltip 等通用规则应引用 `01.视觉语言 Design Language/`。

## 2. 写作目标

每个图表组件文档应同时服务三类任务：

| 任务 | 文档需要提供的信息 |
|---|---|
| 选型判断 | 图表能力、适用场景、不适用场景、数据条件 |
| 设计生成 | 视觉构成、布局关系、组件依赖、交互规则 |
| 代码生成 | 数据字段映射、G2 / ECharts 配置映射、输出约束 |

图表组件文档不是图表百科，也不是示例截图说明。所有内容都应转化为可执行规则。

## 3. 标准章节

图表组件文档应优先使用以下结构：

```txt
1. 文档定位
2. 图表定义
3. 属性摘要
4. 适用场景
5. 不适用场景
6. 数据结构
7. 视觉构成
8. 设计原则
9. 交互规则
10. 与视觉语言的关系
11. G2 / ECharts 映射
12. AI 输出格式
13. 禁止项
14. AI 自查清单
```

当目标图表非常简单时，可以合并章节，但不得省略：

- 图表定义。
- 适用场景。
- 不适用场景。
- 数据结构。
- 视觉构成。
- 设计原则。
- 交互规则。
- 禁止项。
- AI 自查清单。

## 4. 属性摘要写法

属性摘要用于让 AI 快速建立图表能力边界。

建议使用表格：

| 属性 | 写法要求 |
|---|---|
| 图表类型 | 使用稳定分类，如基础图表、统计图表、复合图表 |
| 图形形态 | 描述主要视觉编码，如线条、柱体、点、面积、节点、边 |
| 核心功能 | 描述图表主要分析能力，如趋势、比较、分布、关系、构成 |
| 适合的数据 | 写清字段类型和字段数量 |
| 数据与图形映射 | 明确字段如何映射到坐标、颜色、大小、形状等 |
| 数据体量限制 | 给出可执行上限或判断规则 |

不要只写“适合大量数据”“适合少量数据”这类模糊描述。

## 5. 适用场景写法

适用场景应围绕业务问题，而不是围绕图形外观。

推荐句式：

```txt
当用户需要判断 [业务对象] 在 [连续时间 / 有序维度 / 分类维度] 上的 [变化 / 比较 / 分布 / 关系] 时，优先使用该图表。
```

每个适用场景应包含：

- 分析目标。
- 数据条件。
- 图表价值。

例如折线图的适用场景不应写成“适合画线”，而应写成“适合观察指标随时间变化的趋势、波动和异常点”。

## 6. 不适用场景写法

不适用场景用于避免 AI 误选图表。

每条不适用场景应说明：

- 为什么不适合。
- 应考虑的替代方向。

例如：

| 不适用条件 | 原因 | 替代方向 |
|---|---|---|
| 无序分类比较 | 会暗示不存在的连续关系 | 柱状图、条形图 |
| 部分与整体构成 | 线条不表达占比结构 | 饼图、环图、堆叠图 |
| 强调精确单值 | 趋势图不适合作为数值卡片 | 指标卡、表格 |

## 7. 数据结构写法

数据结构必须写清字段角色，而不是只给示例数据。

建议包含：

| 字段角色 | 是否必需 | 说明 |
|---|---|---|
| 维度字段 | 必需 | 通常映射到横轴、分组、节点或类别 |
| 指标字段 | 必需 | 通常映射到纵轴、大小、长度或角度 |
| 分组字段 | 可选 | 用于多系列、多类别对比 |
| 标记字段 | 可选 | 用于异常、阈值、目标值、阶段等辅助信息 |

如果图表有明确数量限制，应写成确定规则，例如“同一图中系列数不建议超过 5 个”。

## 8. 视觉构成写法

视觉构成应从“数据如何被编码”出发描述。

建议写法：

| 构成元素 | 作用 | 是否必需 |
|---|---|---|
| 坐标轴 | 承载维度与指标刻度 | 视图表类型决定 |
| 图形主体 | 承载核心数据编码 | 必需 |
| 数据点 / 节点 | 表示单个数据位置 | 视图表类型决定 |
| 图例 | 区分多系列或类别 | 多系列时必需 |
| 标签 | 标记关键值或辅助解释 | 可选 |
| Tooltip | 展示悬停时的精确值 | 交互图表建议启用 |

不要把截图中的装饰层、示意标注或 Figma 容器当作图表构成。

## 9. 设计原则写法

设计原则应写成可判断的规则。

每条设计原则建议包含：

- 原则名称。
- 适用条件。
- 执行方式。
- 反例风险。

例如：

```txt
克制系列数量：当图表用于趋势判断时，应控制同屏系列数量，避免多条线相互遮挡；如果系列过多，应改用筛选、分面或小多图。
```

避免写成“保持美观”“适当突出”等无法执行的描述。

## 10. 交互规则写法

交互规则应说明用户动作、系统反馈和启用条件。

建议表格：

| 交互 | 启用条件 | 系统反馈 |
|---|---|---|
| 悬停 | 需要读取精确值 | 显示 Tooltip、辅助线或高亮点 |
| 缩放 | 数据点密集或时间跨度长 | 提供 dataZoom、刷选或滚轮缩放 |
| 标记 | 存在阈值、异常、阶段或目标 | 显示辅助线、辅助区间、关键标签 |
| 图例筛选 | 多系列对比 | 支持隐藏、显示或高亮系列 |

交互不是越多越好。只有当它能提升读数、筛选或洞察效率时才启用。

## 11. 与视觉语言的关系

图表组件文档不直接定义通用样式，应引用以下上游文档：

| 视觉规则 | 引用方向 |
|---|---|
| 主题变量 | `01.视觉语言 Design Language/02.主题变量 Theme-token.md` |
| 色彩系统 | `01.视觉语言 Design Language/03.色彩系统 Color.md` |
| 字体排版 | `01.视觉语言 Design Language/04.字体排版 Typography.md` |
| 坐标轴 | `01.视觉语言 Design Language/06.坐标轴 Axis.md` |
| 图例 | `01.视觉语言 Design Language/07.图例 Legend.md` |
| 标签 | `01.视觉语言 Design Language/08.标签 Label.md` |
| Tooltip | `01.视觉语言 Design Language/09.提示信息 Tooltip.md` |

如果图表组件需要特殊样式，应写成“该图表对上游规则的使用方式”，而不是在本文件重新定义样式系统。

## 12. 引擎映射写法

引擎映射只写关键配置关系，不在图表组件文档中展开完整代码。

建议包含：

| 语义 | G2 | ECharts |
|---|---|---|
| 图表类型 | mark 或 geometry | `series.type` |
| 横轴字段 | `encode('x', field)` | `xAxis` + `encode.x` |
| 纵轴字段 | `encode('y', field)` | `yAxis` + `encode.y` |
| 分组字段 | `color` / `seriesField` | 多 series 或 `encode.seriesName` |
| 交互 | interaction | `tooltip`、`dataZoom`、`legend` |

完整实现细节应交给 `04.引擎适配 Adaptation/`。

## 13. 禁止项

图表组件文档不得：

- 只描述图形外观，不说明数据条件。
- 只给截图，不提炼规则。
- 使用模糊区间值作为规范。
- 复制 Figma 大段原文而不转化。
- 在单个图表文档中重新定义全局颜色、字体、坐标轴和图例规范。
- 把“可以实现”写成“推荐使用”。
- 忽略不适用场景和数据体量限制。

## 14. AI 自查清单

生成或维护图表组件文档前检查：

- 是否说明该图表解决的核心分析问题？
- 是否写清适用场景和不适用场景？
- 是否写清字段角色和数据限制？
- 是否把视觉构成拆成数据编码元素？
- 是否引用而不是复制视觉语言规则？
- 是否给出必要交互及启用条件？
- 是否补充 G2 / ECharts 的语义映射？
- 是否有明确禁止项？
- 是否避免了模糊词和不可执行描述？
